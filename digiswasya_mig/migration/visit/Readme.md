# Visit Migration вАФ Narsala (appdetails_id = 42)

## Overview

This script migrates `HealthCase_healthcase` records from the legacy DigiSwasthya database
to the `visit` table in the new DB, scoped only to **appdetails_id = 42 (DS-TMC-009 / Narsala)**.

**Script:** `migrate_visit_appid42.py`
**Run after:** `migrate_patient_appid42.py` must be completed first.

---

## How to Run

```bash
# Preview only вАФ no DB writes
python migrate_visit_appid42.py --preview

# Migrate first 10
python migrate_visit_appid42.py --limit 10

# Migrate ALL
python migrate_visit_appid42.py
```

## Status Mapping

Legacy `HealthCase_healthcase.status` is an **integer**.
New DB expects a **string** (USER-DEFINED enum type).

| Legacy Value | Assumed Meaning | New DB Value |
|---|---|---|
| `0` | New / Open case | `ACTIVE` |
| `1` | In progress | `IN_PROGRESS` |
| `3` | Closed / Done | `COMPLETED` |

>  **TODO:** These mappings are assumed based on common patterns.
> Please confirm what `0`, `1`, `3` mean in the legacy DigiSwasthya app
> and update `STATUS_MAP` in the script if wrong.
> If an unknown value is encountered, it defaults to `ACTIVE` with a warning log.

## Vitals JSON Mapping

Vitals are embedded inside `HealthCase_commentshealthcase.commenttext` as a JSON string.
The **latest comment per healthcase** is used (ordered by `datetime DESC`).

| New DB Key (`vitals_json`) | Legacy JSON Key in `commenttext` | Notes |
|---|---|---|
| `bp` | `"BP"` | String like `"125/84"`, stored as-is |
| `spo2` | `"Spo2 "` | Note the **trailing space** in legacy key |
| `pulse` | `"Pulse"` | Stored as integer |
| `temp_f` | `"Temperature"` | Stored as number |
| `height_cm` | `"Height"` | Stored as number |
| `weight_kg` | `"Weight"` | Stored as number |
| `sugar_mg_dl` | вАФ |  No legacy source, always `None` |

### Example Legacy `commenttext`:
```json
{
  "Height": 157,
  "Weight": 86,
  "BP": "125/84",
  "Pulse": 87,
  "Temperature": 98,
  "Spo2 ": "99",
  "Do you consume alcohol?": "No",
  "Do you have asthma?": "No",
  ...
}
```

### Resulting `vitals_json` in new DB:
```json
{
  "bp": "125/84",
  "spo2": 99,
  "pulse": 87,
  "temp_f": 98,
  "height_cm": 157,
  "weight_kg": 86,
  "sugar_mg_dl": null
}
```

> Note: The same `commenttext` is also used for `history_notes` (stored as raw string)
> and was used during patient migration for `health_history_json`.

## Patient ID Resolution

The link between legacy healthcase and new DB patient:

```
HealthCase_healthcase.person_id
        вЖУ
patient.legacy_id  (set during patient migration)
        вЖУ
patient.id  (UUID used as visit.patient_id)
```

If a healthcase's `person_id` has no matching `patient.legacy_id` in new DB
(e.g. patient was skipped during migration due to duplicate or validation failure),
the healthcase is **skipped** with a warning log.

---

## Doctor / User Resolution

```
HealthCase_healthcase.doctor_id
        вЖУ
user.legacy_id  (set during user migration)
        вЖУ
user.id  (UUID used as created_by_user_id and assigned_doctor_user_id)
```

> **Current state:** `doctor_id` is NULL for most Narsala healthcase rows.
> Both `created_by_user_id` and `assigned_doctor_user_id` will be NULL for those rows.
>
> If user migration is done later, the `load_doctor_id_to_user_uuid()` function
> will automatically pick up the mappings вАФ no code change needed,
> just re-run the migration.

---

## Skip Logic

Each healthcase row is skipped if:

1. **Already migrated** вАФ `legacy_id` already exists in new DB `visit` table (safe to re-run)
2. **No patient found** вАФ `person_id` has no matching `patient.legacy_id` in new DB
