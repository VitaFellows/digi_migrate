# Prescription Migration ŌĆö Narsala (appdetails_id = 42)

## Overview

This script migrates `BodyVitals_prescription` records from the legacy DigiSwasthya database
to the `prescription` table in the new DB, scoped to prescriptions whose healthcase
belongs to **appdetails_id = 42 (DS-TMC-009 / Narsala)**.

**Script:** `migrate_prescription_appid42.py`

---

## Critical: appdetails_id Mismatch

> ŌÜĀ’ĖÅ The prescriptions for Narsala patients have `appdetails_id = 23` in
> `BodyVitals_prescription`, NOT `42`.

This is because `appdetails_id` in the prescription table refers to the **doctor's app**,
not the **center's app**. The link to Narsala is through `healthcase_id`:

```
BodyVitals_prescription.healthcase_id
        Ōåō
HealthCase_healthcase.id  WHERE appdetails_id = 42
```

**Confirmed by query:**
```sql
SELECT DISTINCT p.appdetails_id, COUNT(*)
FROM "BodyVitals_prescription" p
INNER JOIN "HealthCase_healthcase" hc ON hc.id = p.healthcase_id
WHERE hc.appdetails_id = 42
GROUP BY p.appdetails_id;
-- Result: [(23, 2108)]
-- All 2108 Narsala prescriptions have appdetails_id = 23
```

So both SQLs filter using a subquery instead of direct `appdetails_id` filter:
```sql
WHERE p.healthcase_id IN (
    SELECT id FROM "HealthCase_healthcase"
    WHERE appdetails_id = 42
)
```

---

## Run Order (Prerequisites)

Must run in this exact order:

```
1. migrate_patient_appid42.py     ŌåÉ patient must exist first
2. migrate_visit_appid42.py       ŌåÉ visit must exist second
3. migrate_prescription_appid42.py ŌåÉ this script runs last
```

## Visit ID Resolution

```
BodyVitals_prescription.healthcase_id
        Ōåō
visit.legacy_id  (set during visit migration ŌĆö visit.legacy_id = HealthCase_healthcase.id)
        Ōåō
visit.id  (UUID used as prescription.visit_id)
```

Pre-loaded once before the loop:
```python
{ healthcase_id_str ŌåÆ visit_uuid }
```

If no matching visit found ŌåÆ prescription is **skipped** with a warning log.
This happens when the visit was skipped during visit migration (e.g. patient not migrated).

## Doctor Resolution

```
BodyVitals_prescription.doctor_id
        Ōåō
Doctor_doctordetails.doctorid ŌåÆ name + phone
        Ōåō
new DB user table ŌåÆ match by (name.lower(), phone_last10) WHERE role = 'DOCTOR' AND is_active = TRUE
        Ōåō
user.id ŌåÆ prescription.doctor_user_id
```

Two lookups pre-loaded once before the loop:

**From legacy DB:**
```python
{ doctor_id ŌåÆ (doctor_name, doctor_phone) }
# Source: Doctor_doctordetails
```

**From new DB:**
```python
{ (name_lower, phone_last10) ŌåÆ user.id }
# Source: user WHERE role = 'DOCTOR' AND is_active = TRUE
# name = LOWER(TRIM(first_name || ' ' || last_name))
# phone normalized to last 10 digits
```

## Skip Logic

Each prescription row is skipped if:

1. **Already migrated** ŌĆö `legacy_id` already exists in new DB `prescription` table (safe to re-run)
2. **No visit found** ŌĆö `healthcase_id` has no matching `visit.legacy_id` in new DB

## Fields Left NULL

| Field | Why NULL | What's Needed To Fill It |
|---|---|---|
| `confirmed_diagnosis` | No separate confirmed diagnosis in legacy ŌĆö only one `diagnosis` column | Could be set equal to `provisional_diagnosis` post-migration if needed |
| `is_referral_recommended` | No legacy source, always `False` | Would need manual update or new business logic |
| `referral_specialist_name` | No legacy source | No equivalent in legacy app |
| `deleted_at` | No soft deletes in legacy | Will remain NULL |
